云计算

云计算概念

  1. 发展现状

    1. 概念普及度——云计算的概念

      1. 云计算是一种基于互联网的计算方式，通过这种方式，共享的软硬件资源和信息可以按需求提供给计算机各种终端和其他设备。——维基百科
      2. 云计算是一种模型，可以实现随时随地、便捷地、按需地从可配置计算资源共享池中获取所需要的资源，资源可以快速供给和释放，是管理的工作量和服务提供者的介入降低到最少。——NIST
      3. 云计算是一种计算方式，能通过Internet技术将可扩展的和tanxingdeIT能力作为服务交付给外部用户——Gartner
      4. 云计算是通过网络按需提供可动态伸缩的廉价计算服务。——刘鹏

    2. 云计算的参与者——土地、电力、房子、空调、IT设备

  2. 计算模式的演进：主机系统与集中计算->效用计算->客户机/服务器模式->集群计算->服务计算->个人计算机与桌面计算->分布式计算->网格计算->软件即服务->云计算

    1. 效用计算：考虑购买主机成本高，将IT资源包装成可以度量的服务提供。
    2. 客户机/服务器模式：设备放在内部，通过客户及远程连接使用。
    3. 服务计算：将所有应用程序都作为服务提供，而不是购买软件——在线的网页式服务而不是基于软件。
    4. 云计算——增强的效用计算+广泛的服务计算

  3. 传统分布式计算系统

    1. 计算机集群：将多个可信、静态的独立节点连接起来协作完成计算。
    2. P2P网络：每一个节点即是客户端又是服务器，提供部分系统资源。
    3. 计算网络：将多个异构计算机动态、紧耦合的整合在一起完成计算任务。

  4. 传统分布式系统的特点和问题

  5. 云计算的基本想法：一种完美的分布式计算解决方案——对网格的优化。

    1. 大量计算节点，同构可控
    2. 专用内部网络
    3. 通过互联网对外服务（公有云）
    4. 全面虚拟化：计算、存储、网络、桌面
    5. 集中管理：安全、高效

  6. 云计算和网格计算

  7. 云计算的发展与推动力——技术+商业

    1. 持续优化的动力：节能降耗、降低维护成本、提升资产安全、提升信息系统的容灾备份能力。

  8. 特征与分类

    1. 特征：

      1. 现实中云的特点

        • 大

        • 规模动态伸缩

        • 边界是模糊的

        • 飘忽不定，无法也无需确定具体位置

      2. Elastic Computing Cloud

        • 亚马逊弹性计算云的成功

      3. 云计算的特点

        • 超大规模——需求

        • 虚拟化——技术

        • 按需服务——商业模式

        • 通用

        • 高可伸缩

        • 极其廉价

        • 高可靠——要求

    2. 云部署模型

      1. 公有云：由第三方提供者拥有的可公共访问的云环境。
      2. 社区云：类似公有云，但访问被限定为特定的云用户社区。
      3. 私有云：一家组织单独拥有利用云计算技术来访问组织内部的IT资源。
      4. 混合云：有两种或者更多不同云部署模型组成的云环境。

    3. 云服务模型

    4. Iaas

      1. 通过虚拟机方式对外提供计算和储存能力。
      2. 机器享有公网IP，多个虚拟机之间通过网络进行通信；也有内部子网。
      3. 类似租用独立的计算机。
      4. 用户自行解决多台机器之间的协同问题。
      5. 优势：允许用户动态申请和释放资源；按使用量计费；更高的资源使用效率（节能环保）。

    5. PaaS

      1. 提供用户应用程序的开发和运行环境，包括应用编程接口和运行平台等，支持应用从创建到运行整个生命周期需要的各种软硬件资源和工具。

        1. 将IT功能封装（如数据库、文件系统等）
        2. 主要面向软件开发者

      2. PaaS自身负责资源的动态扩展的容错管理。

      3. 用户无需考虑节点间的配合问题。

      4. 用户自主权降低，需按照给定的编程环境和编程模型构建应用。

      5. 类似MPI编程——只是用于解决模型特定的计算问题。

    6. SaaS

      1. 通过Web浏览器或者软件客户端来访问云上的应用。

    8. 云计算的优势

      1. 经济性
      2. 可扩展性和可伸缩性
      3. 虚拟化
      4. 高可靠
      5. 管理简单
      6. 安全

    9. 大数据

      1. 为什么会产生大数据

        1. 数据产生方式的改变
        2. 人类活动越来越依赖数据
        3. 互联网+物联网

      2. 特征

        1. 3V
        2. 4V+1C
        3. 5V

      3. 大数据与云计算：没有大数据，就不需要云计算，没有云计算，就无法处理大数据。

      4. 大数据促进了云计算的产生，激发了云计算的潜力；云计算为大数据的收集、存储、加工和应用提供了更好的支持——关系密切而多样。

         

    10. 云计算的难点

      1. 数据中心安全控制
      2. 构建高度同构化的虚拟资源
      3. 高可用性的实现
      4. 容错
      5. 节能

    11. 风险

      2. 降低了运行管理控制
      3. 有限的可移植性
      4. 多地区法规和法律

    12. 缺陷与未来

      1. 从平台角度看——没有统一的标准

        • 从计算角度看——针对松耦合的数据处理

        • 从数据角度看——管理和分析商业数据

        • 从资源集成角度看——必须集中到云中

        • 从信息安全角度看——暴露、丢失的风险

      2. 未来：云格计算——结合基准SOA

云计算架构

  1. 中央集权架构：计算机刚出现时的首选，所有东西都集中于一台主机。

  2. 客户机/服务器架构

    • 客户端：承载少量计算任务和所有IO任务

    • 服务器：承载主要计算任务

    • 优点：关注点分离，简化软件复杂度、编程模式

    • 缺点：持久链接，系统伸缩能力受限

  3. 中间层架构——简化和提升伸缩能力

    • 将业务逻辑和数据服务分别放在两个服务器上

    • 客户机到中间层为无状态的非持久链接——提升架构弹性

  4. 浏览器/服务器架构

    • 对客户机、中间层架构的扩展——客户端和中间层均改动

    • 客户端：负载进一步缩减，仅显示和运行基于浏览器的脚本程序

    • 服务器：Web服务器层屏蔽各中间件的差异，提供通用的用户访问界面

    • 扩展性高：对客户机性能无要求

    • 对网络性能要求高

  5. C/S和B/S混合架构

    • 没有一种架构能够适用于所有场景

    • 有些服务需要持久的链接（内部用户）

    • 有些仅通过非持久链接即可（外部用户）

  6. 面向服务的架构（SOA-Service-oriented Architecture）

    1. 每个程序制作自己的本职工作，将服务暴露出来供别人使用。
    2. 多个程序通过一个统一的界面协调工作——控制复杂性，更容易管理。
    3. 区别于分布式对象。

  7. 二维视角

    1. 基础设施+应用程序——前端+后端

    2. 云内部、后端

      • 硬件：服务器、存储器、网络交换机等

      • 管理软件：服务的高可用性、可恢复性，数据一致性，应用伸缩性、可预测性，云安全等

    3. 云外部、前端

      • 并发性、协调性、容错性、开放的API格式、开放的数据格式、承载数据密集型计算

  8. 基础架构进一步分层

    1. 虚拟化层
    2. Web服务层
    3. 服务总线层
    4. 客户机用户界面

  9. 逻辑云栈

    1. 基础设施即服务层
    2. 平台即服务层
    3. 应用程序层-软件即服务层

  10. 传统数据中心缺点

    1. 过多的机器
    2. 利用率低
    3. 应用迁移难
    4. 储存需求增长太快

  11. • 软件定义计算：计算节点的虚拟化：虚拟机成为计算调度和管理的单位，不中断服务的前提下动态迁移。

    • 软件定义存储：分离管理接口与数据读写；统一的管理接口与上层管理软件交互；数据读写可以兼容各种不同的链接方式。

    • 软件定义网络：数据平面（数据转发）和控制平面（转发表的设置等）分离；集中控制，分布式转发。

云数据中心

  1. 对网络架构的需求：低成本、高可扩展性、低配置开销、健壮性、节能

  2. FatTree

    特点：

    1. • 消除了树形结构上层链路对吞吐量的限制

      • 为内部节点间通信提供多条并行链路

      • 与现有数据中心网络使用的以太网结构和IP 配置的服务器兼容

      • 但是布线复杂了；扩展时需要重构；受到端口数限制等。

    2. 常用48端口10G交换机，3层结构中支持27648台主机。

  3. VL2特点

    • 扁平寻址，允许服务实例被放置到网络覆盖的任何地方

    • 负载均衡将流量统一分配到网络路径

    • 终端系统的地址解析拓展到巨大的服务器池

  4. 网络部署

    1. 递归层次——解决可扩展性的较好选择
    2. 增加服务器数量就是提高总的递归层次——添加服务器灵活，可增加的数量大；对交换机性能要求低。
    3. 拓扑层数受限于服务器端口数
    4. 服务器使用两个网卡端口：主用端口连接低层网络，备用端口连接高层网络，一半向上一半向下。
    5. 特点：没有明显的瓶颈链路，出现故障时性能优雅下降，服务可用性高；服务器间路径多，探测的通信和计算开销大；硬件要求特殊：端口数目。
    6. OSA (Optical Switching Architecture)基于光交换的数据中心网络体系结构

  5. 集装箱技术

    1. 特点：高密度；模块化；按需快速部署；移动便捷

  6. 自动化管理

    1. 特征：全面的可视性；自动地控制执行；多层次的无缝集成；综合与实施的报告；全生命周期支持。

  7. 容灾备份：通过在异地建立和维护一个备份储存系统，利用地理上的分离来保证系统和数据对灾难事件的抵御能力。

    1. 数据级容灾

      数据级容灾只保证数据的完整性、可靠性和安全性，但提供实时服务的请求在灾难中会中断。

      应用级容灾

      应用级容灾系统能够提供不间断的应用服务，让服务请求能够透

      明地继续运行，保证数据中心提供的服务完整、可靠、安全。

    3. 关键技术：

      1. 核心：复制技术
      2. 远程镜像技术
      3. 快照技术
      4. 基于IP的SAN远程数据容灾备份技术
      5. 数据库复制技术

虚拟化技术

  1. 虚拟化是云计算的核心技术不是专用技术

    1. 本质：虚拟化的本质就是通过添加一个虚拟化层将原先的物理设备进行逻辑化，转化为一个文件夹或文件，实现软硬件的解耦。
    2. 核心思想是利用软件或固件管理程序构成虚拟化层，把物理资源映射为虚拟资源。在虚拟资源上可以安装和部署多个虚拟机，实现多用户共享物理资源。
    3. 在操作系统中插入一个虚拟化层

  2. 服务器虚拟化的层级

    1. 硬件虚拟化/系统虚拟化

      1. 添加一个成为虚拟化层的软件，管理客户操作系统，使其能够独立与主机操作系统同时运行在同一个硬件上。

      2. 要求

        1. 高效：运行在该虚拟硬件环境中的程序性能应该损失较低。
        2. 隔离：系统资源应该处于VMM的完全控制中。
        3. 复制：为程序提供与原始硬件基本一致的环境。

      3. 优势

        1. 封装性：以虚拟机为粒度。
        2. 多实例：提高计算机性能利用率。
        3. 隔离：独立的运行
        4. 硬件无关性：屏蔽硬件差异。
        5. 特权功能

    2. 操作系统级虚拟化

      1. 操作系统级虚拟化没有虚拟监视器，但是需要一个主机操作系统，在主机操作系统上插入一个虚拟化层——同一个OS内核里运行多个虚拟机。
      2. 优点：最小的启动/停止开销，资源需求低，可扩展性强；可同步虚拟机与宿主操作系统环境状态的变化；允许虚拟机中进程访问尽可能多的主机资源。
      3. 缺点：同一个容器中的OS级虚拟机必须使用相同的客户及操作系统；资源隔离性差。

    3. 云数据中心的虚拟化

      1. 服务器虚拟化；储存虚拟化；网络虚拟化
      2. 目的：提高资源共享与复用，降低成本；提升可扩展性，提升储存资源利用率；方便迁移。

    4. 网络虚拟化

    5. 虚拟机迁移

      1. 需求：物理服务器负载处于动态平衡，当无法提供额外的物理资源时，为了负载均衡需要讲占用热门物理资源的虚拟机迁移到其他物理服务器。同时物理服务器需要定期升级，升级前将虚拟机迁移到其他服务器升级完成后再迁移回去。

      2. 分类

        1. 物理机到虚拟机：P2V；
        2. V2V
        3. V2P

      3. 最困难——内存迁移

      4. 网络迁移：虚拟机所有网络设备包括协议和IP地址都迁移——通过发送并修改相关配置。

      5. 储存设备迁移：迁移数据存储需要占用大量时间和网络带宽，通常的解决办法是以共享的方式共享数据和文件系统，而非真正迁移。目前大多数集群使NAS（Network Attached Storage，网络连接存储）作为存储设备共享数据。

OpenStack

  1. openstack是开源云计算平台，可控制整个数据中心的大型计算、储存和网络资源池。

    1. 管理员通过Web、命令行、Api管理资源；用户通过Web界面使用资源。
    2. 自身不提供虚拟化技术
    3. 调用多种技术实现多资源池管理
    4. 对外提供统一管理接口

  2. 云操作系统指构架于服务器、存储、网络等基础硬件资源和单机操作系统、中间件、数据库等基础软件之上，管理海量的基础硬件、软件资源的云平台综合管理系统。

    1. 特点：网络化；安全；计算的可扩充性。
    2. OpenStack的定位：云计算系统的控制面

  3. OpenStack是一个管理计算、存储和网络资源的数据中心云计算开放平台，通过一个仪表板，为管理员提供了所有的管理控制，同时通过Web界面为其用户提供资源。

  4. 设计理念：开放、灵活、可扩展

  5. 框架

    1. 组件

      1. Nova：计算服务
      2. Swift：对象储存服务
      3. Glance：镜像服务

    2. 其他部件

      1. Keystone：认证管理服务
      2. Cinder：块储存服务
      3. Manila：文件共享储存服务
      4. Neutron：网络服务
      5. Telemetry：计量服务
      6. Heat：编排服务
      7. Horizon：仪表盘服务
      8. 消息队列
      9. 数据库

    3. 网络服务是Openstack服务中最复杂的。

      1. 虚拟网络类型：外部网络，租户网络

  6. 物理部署

    1. 五种节点：数据各个系统的不同组件；三种网络：不同使用目的，Openstack内部网络、外部网络、虚机网络。

  7. DevOps

    1. 要求：软件和部署云基础架构的持续增长的复杂性必须被简化。

    2. 方法：基础架构中的所有内容都自动化；将Openstack分解为多个部分。

    3. 要点：

      1. 简化和模块化Openstack服务；像开发构建一样开发Openstack
      2. 在不影响整个系统的情况下，促进服务的定制和改进
      3. 使用正确的工具来构建服务，确保服务在相同输入情况下输出相同结果
      4. 将服务愿景从如何做切换到要做什么

  8. Ansible

  9. 物理部署

    1. 无中心结构
    2. 节点无本地状态
    3. 多控制器：请求聚合+负载均衡
    4. 水平可扩展：控制节点、计算节点

  10. Keystone：身份认证和服务目录，所有其他服务需要想Keystone注册API节点。

  11. Nova：负责Openstack中提供计算资源服务的项目，最核心的项目

    1. 负责虚拟机生命周期管理，其他计算资源的生命周期管理。

  12. 计算节点——Hypervisor

    1. 计算节点运行nova-compute服务，负责启动和终止虚拟机，通过消息总线监听虚拟机相关请求。

  13. 计算节点——Magnum

  14. 计算节点——Nova

    1. nova单元：按照计算节点为最小单位的管理视角。
    2. 所有计算节点都需要消息总线和数据库服务器通信，随着计算节点增多，使得消息队列和数据库过载，因此产生计算单元。

  15. 储存节点——Swift

    1. 数据与其副本作为二进制大对象储存在对象储存服务器上。
    2. 对象储存没有文件储存的层次结构，对象储存在扁平的命名空间——一种特殊的文件系统。
    3. 不适用于高性能要求以及经常更改的机构化数据。

  16. 存储节点——Cinder

    1. 为云平台提供统一接口，按需分配的，持久化的块储存服务。
    2. 核心功能是对卷的管理，允许对卷等操作。
    3. 为后端不同的储存设备提供了统一的接口。
    4. 块存储作为虚拟机硬盘使用时，必须先分区、创建文件系统，再挂载到虚拟机文件系统层次结构中。

  17. 网络节点——Neutron

    1. 使得运营商能够构建和管理具有所有必要元素的完整网络拓扑。

    2. 核心插件：创建和管理网络、端口和子网

    3. 服务插件：实现高阶网络服务

    4. 代理：部署在计算和网络节点上，和服务器交互。——L2代理、DHCP代理

    5. 虚拟VLAN实现

      1. 核心插件处理虚拟网络和端口的创建。

      2. 虚拟网络创建

        1. VLAN方式：编排连接计算节点和网络节点的物理交换机路径
        2. 隧道网络：将二层数据包封装在IP数据包中；IP网络是已经封装的网络流量传输者

分布式系统

  1. 集中式计算：完全依赖一台大型的中心计算机的处理能力——主机。与其连接的终端设备各具有不相同的、较低的算力。

  2. 分布式计算：多个通过网络互联的计算机都具有一定的算力，他们之间互相传递数据实现信息共享，写作完成一个任务。

    1. 优势：稀有资源共享，在多台计算机上平衡负载，将程序放在最适合他的计算机上运行。

  3. 分布式系统：将海量的计算能力才能处理的问题拆分成许多的小块分配给一套系统中的不同计算节点，最后将分开计算的结果合并成最终的结果。

  4. 分布式计算的一般步骤：设计分布式计算模型；分布式任务分配；编写并执行分布式程序

    1. 【难点：计算任务划分 + 多节点通信】
    2. 如何将复杂算法优化分解为适用于多个节点分别计算的小任务：特别是希望节点之间互不相干
    3. 多数还是要相互通信的：消息队列；分布式存储

  5. ACID原则：原子性、一致性、独立性、持久性

    1. CAP理论：一个分布式系统最多能够同时满足一致性、可用性、分区容错性中的两项。
    2. 对于大型网络来说节点故障和网络出错是常态，一般均以保证AP为主。

  6. BASE理论：追求最终一致性

    1. Basically Available：基本可用-系统出现故障时，允许损失部分可用性，保证核心可用
    2. Soft State：软状态：允许系统存在中间状态，但中间状态不会影响系统的整体可用性
    3. Eventual Consistency：最终一致性-所有数据副本经过一定时间后，能最终达到一致的状态

  7. 一致性算法

    1. 在一个分布式数据库中，如果个节点的初始状态一样，在每个节点执行相同的操作最后能得到一致的状态。

    3. 一致性Hash

      1. 如何高效管理分布式集群，保证高容错率和高可扩展性
      2. 初始化：计算服务器在环上位置；
      3. 使用：计算数据在环上位置。

  8. 分布式系统的特性

    1. 容错性
    2. 高可扩展性
    3. 开放性
    4. 并发处理能力
    5. 透明性

  10. 分布式系统案例

    1. 网格系统——Grid
    2. P2P系统。优点：部署门槛低，增长速度快，容错性高，资源的丰富性和多样性高。应用：共享和分发文件，流媒体，网络电话，志愿计算。
    3. 区块链

  11. Hadoop

    1. Yarn做分布式资源管理；HDFS做分布式存储，MapReduce做分布式计算。
    2. 冗余储存提高了分布式储存的可靠性和分布式计算的可靠性。

  12. Spark：没有类似HDFS的分布式文件系统，但是有一套分布式内存管理系统——核心

    1. 一体化、多元化的大数据处理体系。

    2. 易用、强大、通用

    3. Spark建立在RDD（Resilient Distributed Datasets，弹性分布式数据集）之上的，RDD似的Spark可以用一致的方式处理大数据的不同应用场景。

    4. RDD是一个容错的、并行的逻辑数据结构。

    5. RDD五大特性

      1. 分区猎豹
      2. 依赖列表
      3. 计算函数
      4. 可选分区
      5. 可选优先位置
